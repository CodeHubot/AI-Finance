SSE 技术原理

  • 服务器保持 HTTP 连接不断开
  • 大模型每生成一个 token（约1-2个字）立刻推送
  • 前端实时拼接，用户即时看到内容
  • 感知等待时间从「10秒」变为「0.3秒」
SSE（Server-Sent Events）是现代 AI 应用的标配技术，ChatGPT、Claude 都用这个方式实现打字机效果
🎯

工程意义：相同的 AI 能力，流式输出让用户体验提升 5 倍以上。这也是为什么 AI 产品要专门设计流式 API 而不是等待全量结果。

案例 2 · 核心技术

让 AI 输出结构化数据而非自然语言

❌ AI 输出自然语言（不可用）

"贵州茅台2022年营收约1275亿元，同比增长约16.4%，净利润627亿元，净利率接近50%……"

→ 人能读，机器读不了，无法自动画图
✅ AI 输出 JSON（机器可用）
{
  "analysis_type": "trend",
  "companies": ["贵州茅台"],
  "chart_config": { ...ECharts配置... },
  "chart_title": "营收趋势图"
}
→ 前端直接解析，立刻渲染为图表

实现方式：强制 JSON 模式

调用 API 时加一个参数：

response_format = {"type": "json_object"}

大模型保证 100% 输出合法 JSON，不产生任何额外文字

💡 工程意义

结构化输出让 AI 从「聊天工具」变成「可调用的分析引擎」。这是构建 AI 应用（而非 AI 聊天）的关键技术。

案例 3 · 核心技术

并发 API 调用：三步并行，而非串行等待

❌ 串行方式（慢）
产业链
15s
→
信息摘要
12s
→
评分
10s
=
总计 37s
✅ 并发方式（快）
产业链 15s
┐
信息摘要 12s
┤ 同时发出
评分 10s
┘
总计： ≈ 15s （最慢那个完成即可）

技术实现：Promise.allSettled

// 三个接口同时发出
const [chain, info, score] =
  await Promise.allSettled([
    fetch('/产业链'),
    fetch('/信息摘要'),
    fetch('/评分')
  ])

即使某一个接口失败，其他两个的结果依然可以使用，系统不会全部崩溃。

⚡

设计原则：能并发的一定要并发。后端 AI 调用通常是最慢的环节，并发可以让整体响应时间缩短 60-70%。

上下文窗口管理

大模型有 Token 上限（通常 4K-128K），传入太多内容会超出限制或成本暴增。

  • 只传关键信息，不传原始数据
  • 用摘要替代全文

工程挑战

AI 金融应用的三大核心挑战

🌀

挑战 1：模型幻觉（Hallucination）

AI 可能自信地给出错误数据。金融场景尤其危险——"招商银行2023年净利润为246亿元"（实际是1466亿元）。

应对方案： RAG 强制引用来源 + 结构化数据优先 + 人工复核关键数字
🔒

挑战 2：数据安全与合规

金融机构的内部数据（客户信息、交易记录、研究报告）不能上传至公共 API。监管要求数据本地化。

应对方案： 私有化部署（Qwen/LLaMA 本地版）+ 数据脱敏 + 权限控制
⚡

挑战 3：回答不稳定、格式不一致

同一个问题每次问，AI 给的格式、长度、侧重点可能都不同。生产环境需要高度一致性。

应对方案： 低 temperature（0.1-0.3）+ JSON 强制输出 + 模板约束 + 输出校验

工程思维

AI 应用工程师的四条核心心法

🧩

大模型是组件，不是解决方案

把它当成一个强大的"文字处理函数"——你给它输入，它给你输出。业务逻辑、数据管理、异常处理、安全控制，都还得自己设计。

📝

Prompt 就是代码，需要工程化管理

提示词需要版本控制，需要 A/B 测试，需要评估指标，需要迭代优化。写好一个 Prompt 和写好一个函数一样需要认真对待。

🔩

结构化输出是 AI 应用的核心技巧

让 AI 输出 JSON，才能被系统消费；输出自然语言，只能给人看。从「聊天工具」到「可调用 API」，这是关键一步。

🛡️

失败是常态，要设计降级策略

API 超时、格式错误、内容违规——每种情况都要有兜底。生产级 AI 应用 80% 的工作量在异常处理，而不是"让 AI 说话"。
